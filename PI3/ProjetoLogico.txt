      PROJETO LÓGICO

1. Conceito Geral

Fase de modelagem intermediária entre o modelo conceitual e o físico: define estruturas abstratas sem amarrar ao SGBD específico 
Serve de base para o modelo físico, guiando a representação final no banco NoSQL.

2. Características Principais do Projeto Lógico NoSQL

2.1 Orientação ao tipo de banco

Seleciona o modelo adequado (document, coluna, chave‑valor, grafo etc.) conforme os requisitos 
Exemplos comuns: documentos (MongoDB), famílias de colunas (Cassandra), grafos (Neo4j), ou chave‑valor (Redis).

2.2 Modelagem baseada em padrões de consulta (Query‑Driven Design)

Inverte a modelagem relacional tradicional: no NoSQL se projeta a estrutura de dados a partir dos queries e workflows da aplicação, priorizando desempenho e acesso rápido

2.3 Desnormalização intencional

Incorpora redundância para evitar joins e minimizar custos de leitura.
Documentos agregados evitam necessidade de múltiplas busca; consistência é mantida via transação única 
Uso cuidadoso de embedding vs referencing conforme cardinalidade, atomicidade e frequência de atualização 

2.4 Definição de chaves e índices

Prioriza atributos usados em pesquisas frequentes: definição de primary key, partition key e secondary indexes é feita com base no consumo da aplicação 

2.5 Flexibilidade e evolução de esquema

Projetos schema‑less permitem evolução gradual: novo campos são adicionados sem downtime.
Estratégias formais de evolução (schema evolution) ajudam a manter consistência em ambientes distribuídos 

3. Do Modelo Conceitual ao Projeto Lógico

3.1 Mapeamento de entidades e relacionamentos

Cada entidade mapeia para collection (documento) ou column family.
Propriedades viram campos, relacionamentos se tornam arrays ou referências 

3.2 Identificação de agregados

Combinar entidades frequentemente acessadas juntas em um mesmo documento reduz latência.
Delimitação de agregados pode se basear em Domain‑Driven Design, simplificando a modelagem lógica .

3.3 Definição do fluxo de dados

Documenta padrões de leitura/gravação, volume, consistência necessária (forte vs eventual consistency) e frequência de atualização 

4. Vantagens e Limitações

Vantagens
Alta escalabilidade horizontal (scale‑out) 
Desempenho otimizado para leitura e gravação considerando uso real da aplicação .
Flexibilidade de esquema, adequação a mudanças de requisitos.

Limitações
Gestão de redundância: dados duplicados exigem lógica de atualização nos documentos espelhados.
Sem integridade referencial “nativa”: relacionamentos precisam ser gerenciados pela aplicação 
Modelagem mais complexa, exige conhecimento de padrões NoSQL 

5. Etapas Recomendadas no Projeto Lógico NoSQL

Coleta de requisitos e identificação de consultas-chave (“Query patterns”).
Mapeamento conceitual → lógico: definir collections, campos e agregados.
Decisão: embed ou reference para cada relacionamento com base em cardinalidade e acesso.
Definição de chaves primárias e índices conforme padrões de pesquisa.
Simulação de workloads para validar estrutura e índices.
Planejamento de evolução de esquema com estratégias formais.

6. Ferramentas e Metodologias de apoio

Ferramentas como ER/Studio suportam projeto lógico com NoSQL (ex: MongoDB) 
Abordagens como “Unified Modelset” integram modelos conceituais, lógicos e físicos, adaptando‑os para diferentes bancos 
Domain‑Driven Design (DDD) pode orientar a definição de agregados no modelo lógico 

Conclusão
O projeto lógico NoSQL é uma fase estratégica onde o foco está na experiência de uso — não na lógica relacional. Ele mapeia entidades, relacionamentos, padrões de acesso e requisitos de consistência para uma estrutura lógica acertada, mantendo-se flexível e orientada a desempenho.
